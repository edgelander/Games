## How to port this work upstream

### Option A — Pull Request (recommended)
Opens a reviewable PR from this fork into the original repo; your partner merges it and their
Vercel redeploys.

1. Open this compare link (it pre-fills a fork → upstream PR):
   `https://github.com/Jshort805/Games/compare/main...edgelander:Games:main`
2. Click **Create pull request**, add a short description, and submit.
3. Your partner reviews and clicks **Merge**. Done — the work is upstream.

### Option B — Patch bundle (no PR / offline handoff)
From a clone of this fork:
```bash
git remote add upstream https://github.com/Jshort805/Games.git
git fetch upstream
# A set of .patch files containing only this fork's commits:
git format-patch upstream/main..main -o ../landgrab-patches
# …or a single bundle file you can hand over:
git bundle create ../landgrab.bundle upstream/main..main
```

## Notes & gotchas (carry these to the target environment)

- The app lives in **`landgrab-collaborated/`**. In Vercel, the project's **Root Directory**
  must be set to that folder (the repo root only holds the old single-file `landgrab.html`).
- **Supabase keys are not in the repo** (`.env*` is git-ignored). Whoever deploys sets, in
  their own Vercel **Environment Variables**, then redeploys:
  - `VITE_SUPABASE_URL` → the **bare** project URL `https://<ref>.supabase.co`
    (no `/rest/v1`, no trailing slash).
  - `VITE_SUPABASE_ANON_KEY` → the Supabase **Publishable** key.
- Run **`landgrab-collaborated/supabase-setup.sql`** once in the target Supabase project to
  create the `plots` table, realtime, RLS policies, and the public `plots` storage bucket.
- It's a **PWA** with **auto-update** (`src/pwa.js`): the service worker self-activates
  (`skipWaiting` + `clientsClaim`), polls for new versions every 60s and on app foreground,
  and reloads to the latest build automatically — no cache clear or re-add needed. The very
  first deploy onto a device still installs normally.
